# 08 — Playlist write-back (bonus plan: insights → your actual Spotify)

> Close the loop: every analytical result — discovery candidates, exploration
> queues, blends — materializes as a real, auto-maintained playlist in the
> Spotify client you already use every day. **Effort: S–M.** Depends on:
> bundled scope re-auth; generators depend on 01/02/06/09 landing (each adds
> one). _Claude-suggested addition._

## Why this plan exists

Every other plan produces **queries you have to remember to run**. The graph's
insights become habit-forming the moment they show up *inside Spotify* as
playlists that quietly refresh: open your phone, and "[SPB] Adjacent
Discoveries" has this week's unknowns waiting. It's also the natural *output*
half of the MCP story (plan 05 = ask; this = act).

## Verified constraints

- Playlist **write** endpoints (`POST /users/{id}/playlists`,
  `POST/PUT/DELETE /playlists/{id}/tracks`) are standard, non-deprecated
  surface — expected fine, verify live in T1 (one create + delete). Scopes:
  `playlist-modify-private` (+ `-public` only if ever wanted) — in the bundled
  re-auth.
- 100 tracks max per add/remove call; playlist description ≤ 300 chars.

## Design

### Managed playlists, never anything else

- Every playlist this system touches is **created by it**, named
  `[SPB] <Generator name>`, and tracked in the graph:
  ```
  (:ManagedPlaylist {spotify_id, generator, params_hash, created_at, last_synced})
  (:User)-[:HAS_MANAGED]->(:ManagedPlaylist)
  ```
- Hard safety rule enforced in code: the sync module refuses to modify any
  playlist id not recorded as a ManagedPlaylist. Your hand-made playlists are
  constitutionally untouchable.
- Description auto-stamped: `Generated by spotify-power-browser · <generator>
  · <date> · do not edit (changes are overwritten)`.

### Sync algorithm (idempotent diff, not recreate)

1. Generator (a named Cypher + params) → ordered target track-id list.
2. GET current playlist items → diff (adds / removes / moves).
3. Apply in ≤100-id chunks; reorder only if the generator declares
   order-significant (discovery: yes; queues: yes; sets: definitely).
4. Update `last_synced`, log a one-line summary. `--dry-run` prints the diff.

### v1 generators (one per landed plan)

| Generator | Source | Cadence |
|-----------|--------|---------|
| `adjacent-discoveries` | plan 01's ranked frontier (top 50, popularity-capped) | weekly |
| `exploration-queue <artist>` | plan 02's unheard-albums queue, flattened | on demand |
| `time-capsule <year>` | plan 09's "what you were into in <year>" | on demand |
| `blend <user_a> <user_b>` | plan 06's bridge/overlap sets | weekly |

CLI: `spb playlist sync [generator] [--dry-run]`; cadence via the host's
scheduler of choice (cron / launchd / a compose sidecar loop — decide in T5;
keep it boring).

## Risks & open questions

- A generator regression could blank a beloved playlist → managed-only rule +
  dry-run default in CLI + the diff log make every change inspectable and
  reversible (the graph knows the previous target list — keep last N target
  snapshots on the ManagedPlaylist node for one-command restore).
- Track availability varies by market — the API silently skips unplayable ids;
  log count mismatches, don't fail.
